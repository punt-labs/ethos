\section{Overview}
% ===================================================================

Ethos unifies a name, email, GitHub handle, writing style,
personality, and talents into a single identity that other tools
optionally read.  It is written in Go and ships as a CLI binary plus
a Claude Code plugin with an MCP server.

Identity data lives in two layers: repo-local (git-tracked via the
\fpath{.punt-labs/ethos/} submodule) and user-global
(\fpath{\~{}/.punt-labs/ethos/}).  The repo layer takes priority for
identity lookup; extensions always resolve from the global layer.

The system serves three roles:

\begin{enumerate}
  \item \textbf{Identity registry} --- CRUD operations on identity
    YAML files with layered resolution across repo-local and global
    stores.  Humans and agents share the same schema.
  \item \textbf{Session roster} --- multi-participant awareness for
    Claude Code sessions.  Tracks who is present (human, primary
    agent, subagents), their personas, and parent-child relationships.
  \item \textbf{Persona animation} --- hooks inject personality,
    writing style, team context, and extension data into Claude Code
    sessions at start, before context compaction, and when subagents
    spawn.  Agent definition files are generated from identity data.
\end{enumerate}

Ethos is a \emph{sidecar}, not a dependency.  Other tools (Vox,
Beadle, Biff) read ethos state from known filesystem paths.  They do
not import ethos.  The file format is the contract.

\section{Design Principles}
% ===================================================================

\begin{enumerate}[label=\textbf{P\arabic*}.]
  \item \textbf{Sidecar, not dependency.}  Consumers read files at
    known paths.  No import, no version lock-step, no daemon required.
    Tools adopt ethos at their own pace.
  \item \textbf{Same schema for humans and agents.}  The \field{kind}
    field (\texttt{human} or \texttt{agent}) is the only structural
    difference.  An identity is an identity.
  \item \textbf{Agent definition is a channel binding.}  Like voice
    or email --- the \texttt{.md} file defines tools and workflow,
    ethos defines \emph{who}.
  \item \textbf{Generic extensions, not consumer fields.}  Ethos
    never adds fields for Vox, Beadle, or Biff.  Consumers own their
    own namespace files in the extension directory.
  \item \textbf{Fast.}  Go's $\sim$10\,ms cold start (vs Python's
    $\sim$200\,ms) keeps hook latency invisible.  Ethos is queried on
    session start, on every \cmd{/who}, and before every TTS call.
  \item \textbf{No daemon.}  All concurrency uses \texttt{flock}
    on lock files.  No background process, no database.
  \item \textbf{Preserve identity content.}  Source \texttt{.md}
    files are the authority on personality, writing style, and talent
    content.  Hooks and persona blocks may restructure (strip leading
    headings, fold the first paragraph into the opening line, list
    talents as slugs) but must not discard or summarize the underlying
    meaning.
\end{enumerate}

above it in this ordering.

       mcp  hook  resolve doctor role team attribute
       / \   |  \    |
      /   \  |   \   |

      |

\pkg{cmd/ethos} imports all internal packages directly.
\pkg{internal/mcp} depends on \pkg{identity}, \pkg{session},
\pkg{attribute}, \pkg{role}, \pkg{team}, \pkg{resolve},
\pkg{doctor}, \pkg{hook}, and \pkg{process}.

validation and content resolution.

and \pkg{process}.  \pkg{internal/session}, \pkg{internal/process},
\pkg{internal/role}, and \pkg{internal/team} are leaf or near-leaf
packages.

\end{center}

\subsection{Validation}

\texttt{Identity.Validate()} enforces:

\begin{itemize}
  \item \field{name} and \field{handle} are non-empty.
  \item \field{handle} matches
    \verb|^[a-z0-9]([a-z0-9-]*[a-z0-9])?$| (lowercase alphanumeric,
    may contain internal hyphens but not leading/trailing, used as
    filename).
  \item \field{kind} is one of \texttt{human} or \texttt{agent}.
\end{itemize}

Validation errors are returned as a structured
\texttt{ValidationError} type, not strings.

\subsection{Persistence}

\texttt{Store} takes a root directory path and provides:

\begin{center}
\begin{tabular}{@{}lp{6.5cm}@{}}
\toprule
\textbf{Method} & \textbf{Behavior} \\
\midrule
\texttt{Save(id)}        & Atomic create with \texttt{O\_EXCL} ---
                           rejects duplicate handles.  Validates
                           attribute references.  Creates the
                           extension directory. \\
\texttt{Load(handle)}    & Read YAML, resolve attributes (personality,
                           writing style, talents) to markdown content,
                           assemble extensions from
                           \fpath{<handle>.ext/}.  Pass
                           \texttt{Reference(true)} to skip content
                           resolution. \\
\texttt{List()}           & Return all identities in reference mode
                           (slugs only).  Non-loadable files are
                           reported as warnings. \\
\texttt{FindBy(f, v)}    & Search identities where field \texttt{f}
                           matches value \texttt{v}.  Supported fields:
                           \texttt{handle}, \texttt{email},
                           \texttt{github}.  Returns nil, nil on no
                           match. \\
\texttt{Exists(handle)}   & Check file existence. \\
\texttt{Update(h, fn)}   & Read-modify-write with flock.  Applies
                           mutation function \texttt{fn}, validates
                           attribute references, writes atomically
                           via temp file + rename. \\
\texttt{ValidateRefs(id)} & Check that all attribute slugs
                           (personality, writing style, talents)
                           resolve to existing \texttt{.md} files. \\
\bottomrule
\end{tabular}
\end{center}

\subsection{LayeredStore}

\texttt{LayeredStore} implements \texttt{IdentityStore} with two
layers: repo-local (git-tracked) and user-global.  The repo layer
is optional (nil when not inside a git repository).

\begin{itemize}
  \item \textbf{Load priority}: repo first, then global.  Parse
    errors from the repo layer are surfaced (not silently fallen
    through).  File-not-found falls through to global.
  \item \textbf{Extensions}: always resolve from the global store,
    even when the identity itself is in repo.
  \item \textbf{Attribute resolution}: tries the identity's own
    layer first, falls back to the other layer for any missing
    \texttt{.md} files.
  \item \textbf{List}: returns identities from both stores,
    deduplicated by handle.  Repo wins on collision.
  \item \textbf{Save}: writes to repo if available, otherwise
    global.  Extension directory is always created in global.
  \item \textbf{Update}: updates the identity in the owning store
    (repo if it exists there, otherwise global).  Uses cross-layer
    \texttt{ValidateRefs} so attribute references in either store
    are accepted.
  \item \textbf{Voice migration}: legacy \field{voice} fields in
    repo identities are relocated to the global ext store
    (\fpath{<handle>.ext/vox.yaml}) and stripped from the YAML.
\end{itemize}

\subsection{IdentityStore Interface}

The \texttt{IdentityStore} interface abstracts over both
\texttt{Store} (single-root) and \texttt{LayeredStore} (repo +
global).  All consumers --- MCP handlers, hooks, CLI commands ---
accept \texttt{IdentityStore} rather than a concrete type.

Methods: \texttt{Load}, \texttt{Save}, \texttt{List},
\texttt{FindBy}, \texttt{Exists}, \texttt{Update},
\texttt{ValidateRefs}, \texttt{Root}, \texttt{IdentitiesDir},
\texttt{Path}, \texttt{ExtDir}, \texttt{ExtGet}, \texttt{ExtSet},
\texttt{ExtDel}, \texttt{ExtList}.

\section{Attribute System}
% ===================================================================

Attributes are named markdown documents stored as
\fpath{<kind>/<slug>.md} files under both repo-local and global ethos
roots.  Three kinds are pre-defined:

\begin{center}
\begin{tabular}{@{}llll@{}}
\toprule
\textbf{Kind} & \textbf{Directory} & \textbf{CLI Command} &
  \textbf{Identity Field} \\
\midrule
Talents       & \texttt{talents/}       & \cmd{talent}
              & \field{talents} (list) \\
Personalities & \texttt{personalities/} & \cmd{personality}
              & \field{personality} (scalar) \\
Writing Styles & \texttt{writing-styles/} & \cmd{writing-style}
              & \field{writing\_style} (scalar) \\
\bottomrule
\end{tabular}
\end{center}

\subsection{Kind and Store}

The \texttt{Kind} struct configures a \texttt{Store} for one attribute
type, carrying directory name, CLI command name, display name, and
plural name.  \texttt{Store} provides \texttt{Save}, \texttt{Load},
\texttt{List}, \texttt{Delete}, \texttt{Exists}, and \texttt{Path}.

\texttt{Store} supports an optional fallback store.  When set,
read operations check the fallback (repo layer) first, then the
primary store (global layer).  \texttt{List} merges results from
both layers with the fallback winning on slug collision.  Writes
always go to the primary store.

\subsection{Slug Validation}

Slugs follow the same pattern as identity handles:
\verb|^[a-z0-9]([a-z0-9-]*[a-z0-9])?$|.  Path traversal is
prevented by resolving the candidate path and checking it stays
within the attribute directory.

\section{Extension System}
% ===================================================================

Consumers of ethos (Beadle, Biff, Vox) store tool-specific attributes
on identities without modifying the ethos schema.  Extensions use a
namespaced, file-per-tool design.

\subsection{Storage Layout}

\begin{verbatim}
~/.punt-labs/ethos/identities/
  mal.yaml                     # ethos owns -- core identity fields
  mal.ext/                     # extension directory
    beadle.yaml                # beadle owns -- GPG key, IMAP config
    biff.yaml                  # biff owns -- preferred TTY
    vox.yaml                   # vox owns -- default mood
\end{verbatim}

Each file is a flat YAML map of key-value pairs.  Ethos never reads
or interprets extension contents except to assemble the merged view
returned by \texttt{Load()} and the \tool{identity} MCP tool.

\subsection{Ownership Rules}

\begin{enumerate}
  \item Ethos creates the \fpath{<handle>.ext/} directory on identity
    creation.  Extensions always live in the global store, even when
    the identity is repo-local.
  \item Each tool manages its own \fpath{<namespace>.yaml} file.
  \item Tools may read/write their file directly (sidecar contract) or
    through the ethos CLI/MCP interface.
  \item Tools must not write to another tool's namespace file.
\end{enumerate}

\subsection{Validation Constraints}

\begin{center}
\begin{tabular}{@{}llr@{}}
\toprule
\textbf{Field} & \textbf{Pattern} & \textbf{Limit} \\
\midrule
Namespace   & \verb|^[a-z][a-z0-9-]*$|    & 32 chars \\
Key         & \verb|^[a-z][a-z0-9_]*$|    & 64 chars \\
Value       & Any YAML scalar             & 4096 bytes \\
Keys per namespace    & ---               & 64 \\
Namespaces per persona & ---              & 32 \\
\bottomrule
\end{tabular}
\end{center}

\subsection{Two Scopes}

Extensions exist at two independent scopes with different types:

\begin{enumerate}
  \item \textbf{Persona-level} (durable) --- files in
    \fpath{<handle>.ext/}.  Flat \texttt{map[string]string} (YAML
    scalars only).  Persist across sessions.
  \item \textbf{Session-participant-level} (ephemeral) --- the
    \field{ext} field on each \texttt{Participant} in the session
    roster.  Typed as \texttt{map[string]any} (arbitrary nested
    values).  Deleted when the session ends.
\end{enumerate}

The two scopes are structurally different.  Persona-level extensions
enforce flat string values; session-level extensions accept arbitrary
YAML structures.  A tool may store durable defaults at the persona
level and ephemeral session state at the participant level.

\subsection{Session Context (DES-022)}

Any extension namespace may include a \texttt{session\_context} key.
During \texttt{SessionStart}, \texttt{PreCompact}, and
\texttt{SubagentStart} hooks, ethos collects all
\texttt{session\_context} values from the agent's persona-level
extensions, joins them, and emits the result alongside the persona
block.  This allows consumers (e.g., Biff, Quarry) to inject
behavioral instructions into the model's context without ethos
understanding the content.

\section{Session Roster}
% ===================================================================

The session roster provides multi-participant awareness for Claude
Code sessions.  It answers two questions: ``Who am I?'' and ``Who is
everyone else?''

\subsection{Data Model}

Stored at \fpath{\~{}/.punt-labs/ethos/sessions/<session-id>.yaml}:

\textbf{Field} & \textbf{Source} & \textbf{Purpose} \\

\field{session}      & Hook input        & Session identifier \\

\field{repo}         & Git remote        & Repository name

\field{host}         & \texttt{os.Hostname()} & Short hostname (no
                                           domain) \\
\field{participants} & Hook lifecycle    & Ordered list of
                                           \texttt{Participant}
                                           records \\

\end{longtable}
\end{center}

Each participant record:

\begin{center}
\begin{longtable}{@{}llp{5cm}@{}}
\toprule
\textbf{Field} & \textbf{Type} & \textbf{Description} \\
\midrule
\endfirsthead
\toprule
\textbf{Field} & \textbf{Type} & \textbf{Description} \\
\midrule
\endhead
\field{agent\_id}    & string  & OS login, Claude PID, or subagent ID \\

                                 \texttt{code-reviewer}) \\

\field{ext}          & \texttt{map[string]any}  & Tool-scoped ephemeral
                                 metadata \\
\bottomrule
\end{longtable}
\end{center}

\subsection{Tree Structure}

Participants form a tree.  The root (no parent) is the human user.
The primary Claude agent is the root's child.  Subagents are children
of the primary agent.

\begin{verbatim}
mal (root -- $USER)
  +-- 19147 (archie -- Claude PID)
       +-- a5734dd (code-reviewer)
       +-- a93c2be (silent-failure-hunter)
       +-- b8823ff (Explore, no persona)
\end{verbatim}

Any participant can derive relationships from the tree: root (walk up
to \field{parent:~empty}), initiator (my \field{parent}), delegates
(anyone whose \field{parent} is my \field{agent\_id}), siblings (same
\field{parent}).

\subsection{Lifecycle via Hooks}

\begin{center}
\begin{tabular}{@{}llp{4.5cm}@{}}
\toprule
\textbf{Hook Event} & \textbf{Roster Operation} & \textbf{Data
  Available} \\
\midrule
\texttt{SessionStart}  & Create roster, join root + primary;
                          emit persona block + team context;
                          generate agent files
                        & \field{session\_id}, \texttt{\$USER},
                          Claude PID \\
\texttt{SubagentStart}  & Join subagent; emit persona if matched
                        & \field{session\_id}, \field{agent\_id},
                          \field{agent\_type} \\
\texttt{SubagentStop}   & Leave subagent
                        & \field{session\_id}, \field{agent\_id} \\
\texttt{PreCompact}     & Re-emit full persona block + team context
                          (survives compaction)
                        & \field{session\_id} \\
\texttt{SessionEnd}     & Delete roster + PID-keyed session file
                        & \field{session\_id} \\
\bottomrule
\end{tabular}
\end{center}

\subsection{Concurrency}

Multiple \texttt{SubagentStart} hooks may fire in parallel.  Roster
writes use \texttt{flock(LOCK\_EX)} on a per-session lock file
(\fpath{sessions/<session-id>.lock}) before read-modify-write.  Roster
files are written atomically via temp file + rename.

\subsection{Session Discovery}

Non-hook callers (Biff, Vox) need the session ID without receiving it
on stdin.  The \texttt{SessionStart} hook writes the session ID to a
PID-keyed file:

\begin{verbatim}
~/.punt-labs/ethos/sessions/current/<claude-pid>
\end{verbatim}

Any descendant process walks the process tree to the topmost
\texttt{claude} ancestor PID, reads that file, and gets the session
ID.  The \texttt{SessionEnd} hook deletes the file.

\subsection{Staleness Detection}

\texttt{Store.Purge()} cleans up rosters from crashed sessions:

\begin{enumerate}
  \item Extract the primary agent's \field{agent\_id} (second
    participant).
  \item If numeric: check whether the PID is alive via
    \texttt{os.FindProcess} + signal~0.  If dead, treat as stale
    immediately (no TTL fallback).
  \item If non-numeric (parse fails): age-based fallback (24-hour
    TTL on the \field{started} timestamp).
  \item Rosters with fewer than 2~participants are always stale
    (malformed).
\end{enumerate}

\texttt{PurgeCurrent()} additionally cleans stale PID-keyed files
from the \fpath{sessions/current/} directory when the referenced
process is no longer alive.

\section{Teams and Roles}
% ===================================================================

\subsection{Role Model}

A role is a named YAML file at
\fpath{roles/<slug>.yaml}.

\begin{center}
\begin{tabular}{@{}llp{6cm}@{}}
\toprule
\textbf{Field} & \textbf{Type} & \textbf{Description} \\
\midrule
\field{name}             & string       & Slug-format name \\
\field{responsibilities} & []string     & What the role does \\
\field{permissions}      & []string     & What the role may access \\
\field{tools}            & []string     & MCP tools available to agents
                                          in this role (used in agent
                                          file generation) \\
\bottomrule
\end{tabular}
\end{center}

Role names follow slug validation.  CRUD is provided by
\texttt{role.Store}, with \texttt{role.LayeredStore} searching both
repo-local and global roots.  Deletion is guarded by referential
integrity: a role cannot be deleted if any team references it.

\subsection{Team Model}

A team is a named YAML file at
\fpath{teams/<slug>.yaml}.

\begin{center}
\begin{tabular}{@{}llp{5.5cm}@{}}
\toprule
\textbf{Field} & \textbf{Type} & \textbf{Description} \\
\midrule
\field{name}           & string           & Slug-format name \\
\field{repositories}   & []string         & Repos this team owns
                                            (org/repo format) \\
\field{members}        & []Member         & Identity-role
                                            bindings \\
\field{collaborations} & []Collaboration  & Directed relationships
                                            between roles \\
\bottomrule
\end{tabular}
\end{center}

\textbf{Member}: \field{identity} (handle) + \field{role} (slug).

\textbf{Collaboration}: \field{from} (role) $\to$ \field{to} (role),
with \field{type} constrained to \texttt{reports\_to},
\texttt{collaborates\_with}, or \texttt{delegates\_to}.

\subsection{Z-Spec Invariants}

\texttt{team.Validate()} enforces referential integrity via
callback functions (no circular imports):

\begin{itemize}
  \item Team name follows slug rules.
  \item At least one member.
  \item Every member's identity exists (via \texttt{identityExists}
    callback).
  \item Every member's role exists (via \texttt{roleExists}
    callback).
  \item No duplicate identity/role pairs.
  \item Every collaboration's \field{from} and \field{to} roles are
    filled by at least one member.
  \item No self-collaborations.
  \item Collaboration \field{type} is one of the three valid values.
\end{itemize}

These invariants are checked on team creation and member addition.

\section{Identity Resolution}
% ===================================================================

\subsection{Human Resolution}

\texttt{resolve.Resolve(store, sessionStore)} determines the active
identity handle through a 4-step priority chain.  The chain stops
at the first match:

\begin{enumerate}
  \item \textbf{iam declaration} --- walk the process tree via
    \texttt{FindClaudePID()}, read the PID-keyed session file, load
    the roster, find the matching participant.  If found with a
    persona, return it.  If found with no persona, stop (do not fall
    through --- this is an explicit ``no identity'').
  \item \textbf{git config user.name} --- run
    \texttt{git config user.name}, match against identity
    \field{github} field via \texttt{FindBy("github", name)}.
  \item \textbf{git config user.email} --- run
    \texttt{git config user.email}, match against identity
    \field{email} field via \texttt{FindBy("email", email)}.
  \item \textbf{\$USER} --- read the \texttt{USER} environment
    variable, match against identity \field{handle} field via
    \texttt{FindBy("handle", user)}.
\end{enumerate}

Returns an error when no step matches.

\subsection{Agent Resolution}

\texttt{resolve.ResolveAgent(repoRoot)} reads the repo-local
configuration file.  Tries \fpath{.punt-labs/ethos.yaml} first,
falls back to the legacy path \fpath{.punt-labs/ethos/config.yaml}.
Returns the \field{agent} handle from the config, or empty string
if not configured.

The \texttt{RepoConfig} struct also carries a \field{team} field,
used by \texttt{ResolveTeam()} to determine which team owns the
current repository.

% ===================================================================
\section{Process Tree Walker}
% ===================================================================

\texttt{process.FindClaudePID()} identifies the Claude Code instance
that spawned the current process.  It is used to determine the
primary agent's \field{agent\_id} for session rosters and to locate
PID-keyed session files.

\begin{enumerate}
  \item Read the process table using native OS interfaces:
    \texttt{/proc/<pid>/stat} on Linux,
    \texttt{sysctl kern.proc.pid} on macOS.  No subprocess spawning.
  \item Walk from the current PID upward, following parent pointers.
  \item Track the topmost PID where the command name is
    \texttt{claude}.
  \item Cap the walk at 10~iterations to prevent infinite loops in
    degenerate process trees.
  \item If no \texttt{claude} ancestor is found, fall back to
    \texttt{os.Getppid()}.
  \item Cache the result for the lifetime of the process via
    \texttt{sync.Once} (the answer does not change).
\end{enumerate}

\subsection{Version-Named Binary Handling}

Claude Code installs version-named binaries under
\fpath{\~{}/.local/share/claude/versions/<version>}.  The process
command name for these binaries is the version string (e.g.,
\texttt{2.1.86}), not \texttt{claude}.

\texttt{normalizeClaudeComm(comm, exePath)} detects this pattern:
if the executable path contains \texttt{/claude/versions/}, the
command name is normalized to \texttt{claude}.  On Linux, the exe
path is read from \texttt{/proc/<pid>/exe}.  On macOS, it is read
from \texttt{kern.procargs2}.

\section{Persona Animation}
% ===================================================================

Persona animation is the process of injecting identity context into
Claude Code sessions so the model behaves according to the defined
personality, writing style, and team role.

\subsection{Three Layers}

\begin{enumerate}
  \item \textbf{SessionStart} --- emits the full persona block for
    the repo's default agent.  Includes personality (with opening
    line fold), writing style, talents, extension context
    (\texttt{session\_context} keys), and team context (team members,
    their roles, personalities, collaborations).  Also generates
    \fpath{.claude/agents/<handle>.md} files from identity data and
    installs static agent definitions from the ethos agents
    directory.

  \item \textbf{PreCompact} --- fires before context compaction.
    Re-emits the full persona block, extension context, and team
    context.  This ensures behavioral instructions survive
    compaction.

  \item \textbf{SubagentStart} --- auto-matches the subagent's
    \field{agent\_type} against ethos identities.  If a match is
    found, emits the subagent's persona block and extension context.
    Includes a ``You report to'' line referencing the parent agent
    from the session roster.
\end{enumerate}

\subsection{Persona Block Construction}

\texttt{BuildPersonaBlock(id)} assembles the persona from a resolved
identity:

\begin{itemize}
  \item Opening line: ``You are Name (handle), <first prose sentence
    of personality>.''
  \item Remaining personality content (first paragraph skipped to
    avoid redundancy).
  \item Writing style under \texttt{\#\# Writing Style}.
  \item Talent slugs under \texttt{\#\# Talents} (slug list, not
    full content, to stay within context budget).
\end{itemize}

Leading top-level headings (\texttt{\# Title}) are stripped from
personality and writing style content.  Sub-headings are preserved.

\subsection{Team Context Construction}

\texttt{BuildTeamContext(team, roles, identities, selfHandle)} emits
a team section with per-member entries.  For teammates (not self),
each entry includes full personality content, writing style, role
responsibilities, and talent slugs.  The self entry is compact (role
and responsibilities only) since the full persona block was already
emitted.  Collaborations are listed as compact directed edges.

\subsection{Agent File Generation}

\texttt{GenerateAgentFiles()} creates \fpath{.claude/agents/<handle>.md}
files from ethos identity, personality, writing style, and role data.
Files include YAML frontmatter (name, description, tools from the
role) and a markdown body (opening line, personality, writing style,
responsibilities, talents).  The main agent and human identities are
skipped.  Writes are idempotent: files are only written when content
differs from the existing file.

\section{Design Invariants}
% ===================================================================

These invariants are enforced by the architecture and hold across
all operations:

\begin{enumerate}
  \item \textbf{Sidecar contract.}  No consumer imports ethos.
    The contract is file paths and YAML formats.  Upgrading ethos
    never forces consumer upgrades.

  \item \textbf{Unified schema.}  Humans and agents use the same
    \texttt{Identity} struct.  The \field{kind} field is the only
    difference.  No separate code paths.

  \item \textbf{No consumer-specific fields.}  Ethos will never add
    \field{gpg\_key\_id}, \field{preferred\_tty}, or any field that
    serves a single consumer.  The extension system exists for this.

  \item \textbf{Atomic identity creation.}  \texttt{Store.Save}
    opens the file with \texttt{O\_EXCL}.  Two concurrent creates
    of the same handle: one succeeds, one fails.  No partial writes.

  \item \textbf{Atomic roster writes.}  Roster files are written to
    a temp file, then renamed.  No reader sees a partial roster.

  \item \textbf{Flock concurrency.}  All roster mutations acquire
    \texttt{flock(LOCK\_EX)} on a per-session lock file.  No daemon,
    no database, no race conditions on concurrent subagent hooks.
    Identity updates also use flock on per-identity lock files.

  \item \textbf{Extension isolation.}  Each consumer's extension
    data lives in its own YAML file.  Concurrent writes by different
    tools cannot corrupt each other's data.  Extensions always live
    in global, even for repo-local identities.

  \item \textbf{Process tree caching.}  \texttt{FindClaudePID()}
    caches the result for the process lifetime via
    \texttt{sync.Once}.  The Claude ancestor PID does not change
    during a session.

  \item \textbf{No shell execution.}  All subprocess invocations use
    argument lists, never shell interpretation.

  \item \textbf{Depth-bounded tree walk.}  The process tree walk
    caps at 10~iterations to prevent infinite loops in degenerate
    process tables.

  \item \textbf{Layered resolution.}  Repo-local takes priority for
    identity, attribute, role, and team lookup.  Global is the
    fallback.  Extensions are always global.  This mirrors Git's
    resolution: local config overrides global config.

  \item \textbf{Referential integrity.}  Teams enforce that every
    member's identity and role exist at creation and mutation time.
    Roles cannot be deleted while referenced by any team.
    Collaborations must reference roles filled by current members.

  \item \textbf{Idempotent agent generation.}  Generated agent files
    are only written when their content differs from the existing file
    on disk.  This prevents unnecessary file churn and git noise.
\end{enumerate}

\section{Security Boundaries}
% ===================================================================

\subsection{Filesystem Boundary}

\begin{itemize}
  \item \textbf{User-global state}: \fpath{\~{}/.punt-labs/ethos/} ---
    identities, extensions, sessions.  Standard user-mode file
    permissions.
  \item \textbf{Repo-local state}: \fpath{.punt-labs/ethos/} ---
    identities, attributes, roles, teams, agents.  Git-tracked;
    visible to collaborators.
  \item \textbf{No secrets.}  Ethos stores no passwords, API keys,
    or tokens.  Sensitive bindings (GPG keys, IMAP passwords) live
    in consumer extensions or credential stores, not in ethos.
\end{itemize}

\subsection{Process Boundary}

\begin{itemize}
  \item \textbf{MCP}: stdio transport.  No network listener.
  \item \textbf{Native proc reading}: \texttt{/proc/<pid>/stat} on
    Linux, \texttt{sysctl kern.proc.pid} on macOS.  Read-only.
    Signal~0 for process liveness checks (no side effects).  No
    subprocess spawning for process table queries.
\end{itemize}

\subsection{Trust Boundary}

\begin{itemize}
  \item \textbf{Identity data is self-declared.}  Ethos does not
    verify that the email, GitHub handle, or voice binding is
    authentic.  Verification is the responsibility of consumers
    (Beadle verifies PGP, Biff verifies GitHub tokens).
  \item \textbf{Extension data is opaque.}  Ethos enforces structural
    constraints (namespace format, key format, size limits) but does
    not validate or interpret values.
  \item \textbf{Path traversal prevention.}  Attribute slugs are
    validated against a strict pattern and resolved paths are
    checked to remain within the target directory.  Handle-based
    file paths use \texttt{filepath.Base} to prevent directory
    escape.
\end{itemize}

\section{Key Properties}
% ===================================================================

The architecture guarantees these properties, verifiable by
inspection and testing:

\begin{enumerate}
  \item \textbf{Resolution totality}: \texttt{Resolve()} always
    returns a handle or an error.  There is no silent empty-string
    fallback.

  \item \textbf{Extension namespace isolation}: concurrent writes by
    different tools target different files.  No merge conflicts.

  \item \textbf{Session liveness}: \texttt{Purge()} uses process
    signal~0 for numeric PIDs and age-based TTL for non-numeric IDs.
    \texttt{PurgeCurrent()} cleans stale PID-keyed files.  Every
    stale roster is eventually cleaned up.

  \item \textbf{Hook-driven lifecycle}: the session roster is fully
    managed by six Claude Code hook events.  No polling, no daemon,
    no manual intervention.

  \item \textbf{Cold start performance}: Go binary with no runtime
    dependencies.  $\sim$10\,ms cold start measured on Apple~M2.
    Hooks are invisible to the user.

  \item \textbf{Store injectability}: all CRUD operations accept a
    store with a configurable root path (or an
    \texttt{IdentityStore} interface).  Tests use
    \texttt{t.TempDir()}.  No hardcoded \texttt{\$HOME}.

  \item \textbf{Persona survivability}: the \texttt{PreCompact} hook
    re-injects the full persona block, extension context, and team
    context before context compaction.  Behavioral instructions are
    never lost to compaction.

  \item \textbf{Team integrity}: Z-spec invariants (identity
    existence, role existence, no duplicates, collaboration
    validity) are enforced on every write operation.  A team is
    never persisted in an invalid state.
\end{enumerate}
