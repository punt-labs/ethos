# Writing Agent Definitions

A practical guide for writing effective Claude Code agent definitions with ethos.

## The Three Layers: Persona, Role, Mission

An effective agent has three layers of context, each with a different
lifecycle:

| Layer | What It Provides | Lifecycle | Where It Lives |
|-------|-----------------|-----------|---------------|
| **Persona** | Identity, judgment, taste, communication style | Durable -- survives across tasks and sessions | Ethos identity (personality + writing style + talents) |
| **Role** | Tools, responsibilities, anti-responsibilities, team position | Semi-durable -- changes when the team changes | Ethos role YAML |
| **Mission** | Typed I/O contract: leader, worker, evaluator, write_set, success_criteria, budget | Ephemeral -- one task, then done | Mission contract YAML at `~/.punt-labs/ethos/missions/<id>.yaml`, registered via `ethos mission create` |

The persona stays warm -- it gives the agent judgment, taste, and
voice. The mission stays cold -- it gives the agent precision. The
role sits in between -- it defines what the agent can do and what it
must not do.

Ethos owns persona and role. The leader writes the mission contract
at delegation time and registers it via `ethos mission create`.

## Separation of Concerns

Five sources converge to produce a sub-agent's context:

| Source | Defines | Lifecycle |
|--------|---------|-----------|
| Agent definition (`.claude/agents/<handle>.md`) | Persona + role -- tools, scope, principles, working style | Static, generated by SessionStart |
| SubagentStart hook | Dynamic persona -- personality, writing style, team context, extension `session_context` | Dynamic, injected at spawn |
| CLAUDE.md | Project rules -- standards, workflow, invariants | Loaded via normal message flow |
| Delegation prompt | Freeform leader instructions to the worker; may reference a mission contract by ID | Ephemeral, written by leader per task |
| Mission contract | Typed YAML artifact: leader, worker, evaluator, write_set, success_criteria, budget -- created via `ethos mission create` | Ephemeral, one contract per delegation |

Never duplicate content across these sources. Personality goes in the
identity, not the agent `.md`. Project rules go in CLAUDE.md, not the
agent `.md`. Task-specific constraints go in the delegation prompt,
not the agent `.md`.

When a sub-agent spawns, Claude Code replaces the default system
prompt with the agent definition body. CLAUDE.md loads separately
via normal message flow. The SubagentStart hook injects
`additionalContext` with the persona block, team context, and
extension `session_context`. The delegation prompt arrives as the
task instruction and may direct the worker to load the typed
mission contract via `ethos mission show <id>`. Each source
arrives independently and carries distinct content.

## Agent Definition Anatomy

### Frontmatter

**What ethos generates today** (DES-026):

```yaml
---
name: bwk
description: "Go specialist sub-agent. Implements Go code with tests following Kernighan's principles -- simplicity, clarity, generality."
tools:
  - Read
  - Write
  - Edit
  - Bash
  - Grep
  - Glob
model: "sonnet"              # when role defines a model
---
```

Ethos generates `name`, `description`, and `tools` (from role). The
body includes the personality opening line, personality content,
writing style, talent slugs, and role responsibilities.

**Additional frontmatter fields** (set manually or planned for future
generation):

| Field | Source | Notes |
|-------|--------|-------|
| `name` | Identity handle | Generated. Case-sensitive, must match exactly |
| `description` | Identity + role summary | Generated |
| `tools` | Role definition | Generated. Explicit allowlist -- see Tool Restrictions |
| `model` | Generated (from role, when set) | `sonnet`, `opus`, `haiku`, full ID, or `inherit` |
| `effort` | Manual | `low`, `medium`, `high`, `max` |
| `memory` | Manual | `user`, `project`, or `local` |
| `skills` | Manual / future: generated | Claude Code skills preloaded at startup |
| `hooks` | Generated from `role.tools` | `PostToolUse` hook on `Write\|Edit` runs `make check` for write-enabled roles; review-only roles emit no block |
| `isolation` | Manual | `worktree` for git worktree isolation |
| `maxTurns` | Manual | Cap on agentic turns |
| `background` | Manual | Always run as background task |

### Body

The body has five sections, assembled in this order:

**1. Opening line.** (generated) Generated by ethos from the personality. Format: "You are Name (handle), [role description]." Followed by "You report to [parent] ([parent role])."

**2. Principles.** (generated from personality) 3-5 core principles from the personality. These define how the agent thinks and makes decisions. Drawn from the personality markdown file.

**3. Working Style.** (hand-written) Operational methodology -- how this agent approaches work. This is the primary hand-written section. Covers workflow, tool preferences, testing approach, verification habits.

**4. Responsibilities.** (generated from role responsibilities) Rendered as a `## Responsibilities` section. Concrete list of what the agent owns, one bullet per entry in the role's `responsibilities` list.

**5. What You Don't Do.** (generated from the team graph's `reports_to` edges) Explicit scope boundaries. See Anti-Responsibilities below.

Example (from `bwk.md`):

```markdown
You are Brian K (bwk), a Go specialist on the Punt Labs engineering team.
You report to Claude Agento (COO/VP Engineering).

## Principles

From *The Practice of Programming* (Kernighan & Pike):

1. **Simplicity** -- the simplest solution that works is the best
2. **Clarity** -- write clear code, not clever code
3. **Generality** -- design for the broad case, not the specific one

## Working Style

- Tests first -- write the test, then the code that makes it pass
- Table-driven tests with testify/assert and testify/require
- Errors handled at every call site with context
- Short names for locals, descriptive for exports
- `make check` must pass before you consider anything done
- Race detection mandatory (`-race` flag)

## Responsibilities

- Go package implementation with tests
- code review for Go projects
- adherence to punt-kit/standards/go.md

## What You Don't Do

You report to coo. These are not yours:

- execution quality and velocity across all engineering (coo)
- sub-agent delegation and review (coo)
- release management (coo)
- operational decisions (coo)

Talents: engineering
```

## Anti-Responsibilities

Every agent has an explicit list of things it does not do. These are
derived mechanically from the team graph: for each `reports_to` edge
from the agent's role to a target role, the target role's
`responsibilities` are listed in the generated `## What You Don't Do`
section with parenthetical attribution to the target.

The shape is fixed:

- A section heading `## What You Don't Do` with a blank line below it
- A preamble line naming the target role(s) the agent reports to,
  using Oxford comma for three or more: `You report to coo, ceo, and
  cto. These are not yours:`
- A flat bullet list of the target roles' responsibilities with
  parenthetical attribution on every bullet, regardless of how many
  targets contributed

Degraded cases:

- **Zero `reports_to` edges from the agent's role** — no section is
  emitted. The file ends at `## Responsibilities` (or wherever the
  previous section was).
- **All target roles have empty `responsibilities` lists** — no section
  is emitted.
- **Some targets populated, some empty** — section is emitted; the
  preamble names only the targets that contributed bullets.
- **Missing target role** (team graph references a role that fails to
  load) — the missing target's bullets are absent and a stderr warning
  is emitted at generation time; other targets still contribute.

Scope limits in the current MVP:

- Only the `reports_to` edge type contributes anti-responsibilities.
  `collaborates_with` and `delegates_to` edges from the agent's role
  produce a stderr warning and no bullets. Their semantics (peer,
  downward) deserve different phrasing and are deferred to future beads.
- No schema change: `Role` has no `anti_responsibilities` field. The
  derivation reads the team graph and the target roles' own
  `responsibilities` lists.
- No pattern matching, LLM rewriting, or verb-phrase transformation.
  Responsibility strings are emitted verbatim (after whitespace
  normalization: leading/trailing whitespace is trimmed, embedded
  newlines collapse to spaces, empty-after-trim strings are skipped).

Anti-responsibilities serve two purposes: they prevent the agent from
exceeding its scope, and they tell the agent which role handles the
excluded work so it can escalate rather than improvise.

## Tool Restrictions

Tools come from the role definition (DES-026). The `tools` field in the agent frontmatter is an explicit allowlist.

Common patterns:

| Agent Type | Tools | Rationale |
|-----------|-------|-----------|
| Implementation | Read, Write, Edit, Bash, Grep, Glob | Full file access for code changes |
| Review (read-only) | Read, Grep, Glob, Bash | Can read and analyze, cannot modify |
| Architect | Read, Grep, Glob | Design only -- no execution or file modification |
| Research | Read, Grep, Glob, WebFetch, WebSearch | Can explore, cannot modify |

Never omit the `tools` field. When omitted, the sub-agent inherits all tools from the parent -- including MCP tools. An implementation agent with access to `mcp__github__merge_pull_request` can merge PRs. An explicit allowlist prevents this.

## Baseline Operational Discipline

Sub-agents lose Claude Code's default system prompt entirely. Without a baseline, the agent does not know to use Read instead of `cat`, or to run `make check` after changes.

The baseline-ops skill is shipped in the ethos binary and deployed to
`~/.claude/skills/baseline-ops/SKILL.md` by `ethos seed` (called
automatically by the installer). To reference it, add `skills: [baseline-ops]`
to the agent frontmatter. It is a preloaded Claude Code skill. Note: Claude
Code "skills" (preloaded content injected at startup) are distinct from ethos
"talents" (domain expertise referenced by slug). The skill covers:

- Use dedicated tools: Read over `cat`, Grep over `grep`, Glob over `find`
- No chained Bash commands (`&&`, `||`, `;`, `$()`, `|`, `for` loops)
- Run `make check` after every change, zero violations before returning
- Never commit -- return results to the delegator
- Never run destructive git commands (reset --hard, checkout ., clean -f)
- Read neighboring files before writing new code, follow existing patterns
- No features beyond the spec
- OWASP top 10 awareness, never log secrets
- Progress tracking with TodoWrite (when available)
- Check quarry for existing knowledge before broad searches
- Concise output, no preamble

The agent frontmatter includes `skills: [baseline-ops]`, and Claude
Code injects the full skill content at spawn.

## Output Format Contracts

When agents produce structured output (code review findings, research results, architecture decisions), define the expected format in the agent body. Consistent output enables the delegator to parse results reliably across multiple delegations.

Example for a review agent:

```markdown
## Output Format

Report findings as a numbered list:

1. **[severity]** file:line -- description
   - What: what is wrong
   - Why: why it matters
   - Fix: suggested change (do not apply it)

Severity levels: CRITICAL, WARNING, NOTE.
End with a summary line: "N findings (X critical, Y warning, Z note)."
```

Without a format contract, the same agent produces different output shapes across delegations. The delegator wastes turns reformatting.

## Scope Enforcement

Three levels, from lightest to strongest:

**Level 1: Implicit (role definition).** The "What You Do" section naturally limits scope. An agent described as "Implement Go packages" is unlikely to edit Python files unprompted.

**Level 2: Explicit (anti-responsibilities).** The "What You Don't Do" section names specific actions and redirects to the responsible agent. This handles the common cases where scope is ambiguous.

**Level 3: Critical (repeated enforcement).** For agents operating near domain boundaries -- where the task could reasonably expand -- repeat the constraint multiple times in the body. Example: a security review agent might have "Do not fix code" in both the anti-responsibilities and the output format sections.

Most agents need levels 1 + 2. Level 3 is for agents where experience shows scope creep is a recurring problem.

## Context Hygiene

Sub-agents accumulate context over their lifetime — tool results, file
reads, intermediate notes and summaries. When a sub-agent finishes one task and
is asked to continue with another, the stale context from the first
task can degrade performance on the second.

**Resume when context overlap is high.** If the next task operates on
the same files, the same subsystem, and the same constraints, reusing
the agent avoids re-reading and re-discovering what it already knows.

**Spawn fresh when context has shifted.** If the next task is in a
different domain, different files, or different phase of work, spawn
a new agent with a clean prompt. A fresh agent with a precise prompt
outperforms a reused agent carrying irrelevant context.

The deciding factor is prompt quality, not efficiency. A fresh spawn
costs one agent startup delay. Stale context costs wrong
assumptions, missed constraints, and wasted turns recovering.

## Generated vs Hand-Written

Ethos generates agent definitions from identity data during SessionStart (DES-026). The generated file currently includes:

- **Frontmatter**: name, description, tools (from role), model (when role defines one)
- **Body**: opening line, "You report to" line, remaining personality, ## Writing Style, ## Responsibilities, Talents (inline line)

Note: the "You report to Claude Agento" line is currently hardcoded in
the generator. Future work will source this from the session roster via
SubagentStart instead.

Fields like `model`, `effort`, `memory`, `skills`, and `hooks` are not
yet generated — set them manually or wait for Phase 2 of the
[roadmap](ETHOS-ROADMAP.md). Anti-responsibilities ("What You Don't
Do") are also planned for generation from the team collaboration graph.

The generation is idempotent -- running twice with unchanged identity data produces identical output. Content is compared before writing; identical files are not rewritten.

Hand-written content belongs in these locations:

| Content | Where it lives | Why |
|---------|---------------|-----|
| Personality, temperament | `personalities/<slug>.md` | Ethos generates from this |
| Writing style | `writing-styles/<slug>.md` | Ethos generates from this |
| Domain expertise | `talents/<slug>.md` | Listed as slugs in the agent body |
| Working methodology | Agent `.md` body | Operational, not identity |
| Role constraints | `roles/<name>.yaml` | Drives "What You Do/Don't Do" |
| Project rules | CLAUDE.md | Loaded separately, not in agent |

Best practice: put behavioral content in ethos (personality, writing style, talents). Put operational methodology in the agent `.md` body. Ethos generates the who; you write the what.

## Testing Your Agent

After creating or modifying an agent definition, verify it loads correctly:

```text
Agent(subagent_type="<handle>", prompt="Who are you? What are your principles? What tools do you have?")
```

Check these properties:

1. **Identity**: agent states its name, handle, and role correctly
2. **Personality**: principles and temperament match the personality markdown
3. **Tools**: agent lists the tools from the role, not inherited parent tools
4. **Scope**: agent can articulate what it does and does not do
5. **Operational baseline**: agent knows to use Read over `cat`, run `make check`

If the personality is missing, the SubagentStart hook failed to match the `agent_type` to an identity handle. Verify the `name` frontmatter field matches the identity `handle` exactly (case-sensitive).

## Common Anti-Patterns

**Duplicating CLAUDE.md content.** The agent reads CLAUDE.md separately via normal message flow. Copying project rules into the agent body wastes context tokens and creates two sources of truth that drift.

**Embedding personality in the .md file.** Personality belongs in the ethos identity (`personalities/<slug>.md`). The agent `.md` file is static; ethos hooks inject current personality data at spawn. Hardcoded personality in the `.md` goes stale when the identity changes.

**Omitting tool restrictions.** Without an explicit `tools` field, the sub-agent inherits all parent tools -- including MCP tools for GitHub operations, email, messaging. An implementation agent should not have access to `merge_pull_request`.

**No anti-responsibilities.** Without explicit boundaries, agents drift into adjacent domains. A Go specialist starts making architectural decisions. A reviewer starts applying fixes. State what the agent does not do.

**Monolithic agent definitions.** If the agent body exceeds 300 lines, the agent is trying to do too much. Split into multiple specialized agents with clear scope boundaries. Two 80-line agents outperform one 300-line agent because each maintains focus.

**Duplicating SubagentStart content.** The SubagentStart hook injects the persona block, team context, and extension `session_context` dynamically. Do not hardcode "You report to X" or team structure in the agent body -- it arrives via the hook and stays current.

**Leader as router.** The coordinator should synthesize findings, not relay them. Writing "based on your findings, implement it" pushes understanding onto the worker instead of doing the synthesis yourself. The coordinator reads the research, decides what it means, and writes a precise implementation spec. If the coordinator's prompt to the next agent doesn't demonstrate understanding of the previous agent's output, the coordinator isn't doing its job. Effective delegation is: research agent returns findings → coordinator reads, evaluates, and decides → coordinator writes a spec with file paths, line numbers, and specific changes → implementation agent executes the spec.

**Stale agent reuse.** Reusing a sub-agent across unrelated tasks saves one startup but carries context that confuses the model. See [Context Hygiene](#context-hygiene). When in doubt, spawn fresh.

## Writing Effective Delegation Prompts

The agent definition defines who the agent is and what it can do. The
delegation prompt defines what it should do *right now*. Two artifacts
carry that "right now" payload, and they are not interchangeable:

- The **delegation prompt** is the freeform string the leader passes
  to the `Agent` tool (or equivalent). It is prose. It frames the
  task, recaps prior research, names the bead, and tells the worker
  what to read first. Leadership thinking lives here.
- The **mission contract** is the typed YAML artifact created via
  `ethos mission create --file <yaml>` and stored at
  `~/.punt-labs/ethos/missions/<id>.yaml`. Schema-validated at parse
  time, runtime-enforced by the Phase 3 hooks. Identity, write set,
  success criteria, and budget live here.

Both exist per delegation. The prompt can — and should — reference
the mission ID and tell the worker to load the contract via
`ethos mission show <id>`. The contract pins the typed fields; the
prompt carries the prose context. They reinforce each other.

### Mission Structure

The mission contract is a typed YAML artifact, not freeform prose.
The leader writes it once, registers it via
`ethos mission create --file <path>`, and ethos stores it under
`~/.punt-labs/ethos/missions/<id>.yaml` where the runtime can
enforce it. See DES-031 and the architecture spec's "Mission
Primitive" section for the full runtime model.

```yaml
leader: claude
worker: bwk
evaluator:
  handle: djb
inputs:
  bead: ethos-9ai.x
context: |
  Add Model field to internal/role/role.go and wire through
  GenerateAgentFiles. Default to "inherit" when empty.
write_set:
  - internal/role/role.go
  - internal/role/role_test.go
  - internal/hook/generate_agents.go
  - internal/hook/generate_agents_test.go
success_criteria:
  - Role struct has Model string field with yaml/json tags
  - GenerateAgentFiles emits model in frontmatter when non-empty
  - Default is "inherit" when Model is empty
  - make check passes
  - Tests cover: model set, empty, inherit
budget:
  rounds: 2
  reflection_after_each: true
```

Task framing, constraints, and expected outputs are leadership
*thinking* — they shape the delegation prompt's prose and the
worker's understanding. The contract is the *artifact*: typed
fields the runtime can enforce. The leader's freeform prompt then
tells the worker something like "read mission `m-2026-04-09-001`
via `ethos mission show` before you start."

### Mission Anti-Patterns

**Vague delegation.** "Fix the bug in the auth module" gives the agent
a direction but no contract. Which files? What's the expected behavior?
How will we know it's fixed? The agent fills in the blanks with
guesses.

**Delegating understanding.** "Based on the research findings,
implement the solution" pushes synthesis onto the agent. The mission
should prove the leader understood the input. See the "Leader as
router" anti-pattern above.

**Unbounded scope.** A mission without a `write_set` lets the agent
touch anything. A mission without tight `success_criteria` lets the
agent make design decisions. Both produce work the leader must undo.

### The Persona-Mission Interaction

The persona gives the agent judgment within the mission. When a
mission says "implement function X," the persona determines *how*:
bwk writes it with Kernighan's simplicity principle, rmh writes it
with Hettinger's idiomatic Python. The mission is the same; the
execution reflects the persona.

This is why ethos separates persona from role from mission. The
persona is reusable across thousands of missions. The role is
reusable across a team's lifetime. The mission is single-use.
Conflating them produces agents that are either too rigid (persona
baked into every task prompt) or too generic (no persona at all).
